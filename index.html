Interview Prep

Top 10 Questions — Cloud Security Architect

Q1 How would you design a secure multi-account AWS architecture from scratch? ▶
Start with AWS Organizations — management account only for billing/governance, never workloads. Create dedicated accounts: security (centralized logs, GuardDuty, Security Hub), shared-services, dev, staging, prod. Apply SCPs at OU level to prevent root access, block non-approved regions, and enforce encryption. Use IAM Identity Center (SSO) for all human access — no long-lived IAM users. Centralize CloudTrail logs to security account S3 with bucket policy denying delete. Show you think in terms of blast radius reduction and least privilege by design.
Q2 Explain how you would respond to a compromised EC2 instance in production. ▶
Contain first: remove instance from all security groups (isolate SG with no ingress/egress). Preserve evidence: take EBS snapshot, capture memory if possible, export VPC flow logs for the time window. Revoke credentials: if the instance had an IAM role, revoke all active sessions via a deny-all inline policy. Investigate: CloudTrail for API calls from that role, GuardDuty findings, application logs. Remediate: terminate instance, deploy fresh from golden AMI. Post-incident: update runbook, automate isolation via Lambda for future events.
Q3 What is the difference between a Security Engineer and a Security Architect? ▶
A Security Engineer implements and operates security controls — configures GuardDuty, writes Lambda functions, responds to incidents. A Security Architect designs the security posture — evaluates risk, defines control requirements, creates reference architectures, and ensures that systems are secure by design before they're built. Architects think in threat models and attack paths, not just tool configurations. The key phrase: engineers say "I configured X", architects say "I mitigated Y risk by designing Z control".
Q4 How do you implement Zero Trust in AWS? ▶
Zero Trust means never trust, always verify — identity, device, and context matter for every access decision. In AWS: (1) Identity — all access via IAM Identity Center with MFA, no long-lived keys; (2) Network — private subnets, VPC endpoints instead of internet routing, PrivateLink for AWS services; (3) Workloads — IRSA for pods (no EC2 instance profiles), short-lived credentials everywhere; (4) Data — encrypt at rest + in transit, KMS with key policies tied to identity; (5) Monitoring — CloudTrail + GuardDuty on every action, anomaly detection. The principle: even if an attacker is inside, they can't move laterally.
Q5 Walk me through a STRIDE threat model for a typical 3-tier web application on AWS. ▶
For a web app (ALB → EC2 → RDS): Spoofing — attacker impersonates user (mitigation: Cognito + MFA); Tampering — modify data in transit or at rest (mitigation: TLS 1.2+, KMS encryption, integrity checks); Repudiation — deny performing an action (mitigation: CloudTrail, application-level audit logs); Information Disclosure — S3 public exposure, RDS without encryption (mitigation: bucket policies, encryption, VPC isolation); Denial of Service — flood ALB (mitigation: WAF rate limiting, Shield Advanced); Elevation of Privilege — EC2 role with admin permissions (mitigation: least privilege IAM, SCPs). Each threat maps to MITRE ATT&CK techniques.
Q6 How do you secure a Kubernetes cluster in production? ▶
Layers: API server — private endpoint, RBAC least privilege, audit logging; Authentication — IRSA for pods (no EC2 profiles), short-lived OIDC tokens; Admission control — Kyverno to enforce: no root containers, no privileged mode, image digest required, no latest tag; Runtime — Falco for behavioral detection, seccomp profiles, AppArmor; Network — network policies deny-all by default, only allow explicit flows; Images — Trivy scanning in CI, sign with Cosign, verify in Kyverno; Secrets — External Secrets Operator → AWS Secrets Manager (no secrets in YAML); Node — CIS benchmark hardening, restricted instance metadata.
Q7 What are the most common IAM misconfigurations you see in AWS environments? ▶
(1) Wildcard permissions — Action: "*" or Resource: "*" in IAM policies; (2) Long-lived access keys — human users with static keys instead of role assumption; (3) Overly permissive cross-account trust — trust policy without conditions (no ExternalId, no MFA condition); (4) EC2 instance profiles with admin roles — any exploit on that instance = full AWS access; (5) No SCP guardrails — member accounts can do anything including disabling CloudTrail; (6) PassRole abuse — allowing iam:PassRole without restricting which roles, enabling privilege escalation. Tool to find these: Cloudsplaining, IAM Access Analyzer.
Q8 How do you build a DevSecOps pipeline that doesn't slow down developers? ▶
Key principle: shift left without friction. (1) Fast tools first — linters and SAST run in seconds, don't block for slow scans; (2) OIDC instead of static credentials — no keys to manage; (3) Pre-commit hooks (gitleaks, detect-secrets) — catch secrets locally before they reach CI; (4) Policy as Code in CI — Checkov/OPA give clear, actionable error messages not vague failures; (5) Scan only changed files in PRs, full scan only on merge to main; (6) Developer education — show the exact line and how to fix it; (7) Don't block on LOW/MEDIUM in PRs — only block on CRITICAL/HIGH. Security is a feature, not a gate.
Q9 How would you detect a cryptominer running inside a compromised container? ▶
Multiple signals: (1) Falco rule — detect unexpected outbound connections on port 3333/4444, or spawning of processes like xmrig, minerd; (2) CloudWatch metrics — unusual CPU spike on EC2 node or EKS pod (cryptominer uses ~100% CPU); (3) GuardDuty — CryptoCurrency:EC2/BitcoinTool finding if connecting to known mining pool IPs; (4) VPC Flow Logs — high volume outbound to unknown external IPs; (5) Audit logs — if container was deployed from an unverified image (Kyverno would have blocked this). Prevention: image signing + Kyverno verification means only trusted images run.
Q10 What is your approach to securing AI/LLM workloads on AWS? ▶
OWASP Top 10 LLM as the framework: (1) Prompt injection — Bedrock Guardrails with input filtering, separate system/user context in prompts; (2) Data exposure — RAG access control via IAM-scoped vector DB queries (user can only retrieve what they're authorized to see); (3) Insecure output — output moderation layer, never execute model output directly; (4) Model access control — IAM policies on Bedrock, VPC endpoint for private access, CloudTrail logging all model invocations; (5) Data poisoning prevention — validate RAG data sources, hash verification; (6) Monitoring — log all prompts + responses for anomaly detection. This skill is extremely rare — very few security architects have it.

Zero Trust

3 Pillars — What the €150k Interview Asks

Architect language: Don't say "I configured a firewall". Say "I mitigated lateral movement via L4 segmentation based on STRIDE analysis." These 3 pillars are the exact architecture a Cloud Security Architect must prove — check them off as you implement each.
  • Replace all IAM Users + long-lived access keys with IAM Identity Center (SSO) — federated with Azure AD or Okta
  • Configure OIDC federation between GitHub Actions and AWS — no passwords stored anywhere in the pipeline
  • Apply Permissions Boundaries to cap the maximum privilege any role can ever grant, even if the IAM policy allows more
  • Use IAM Access Analyzer to detect unused permissions and external access automatically
  • No Bastion Hosts with port 22 open — all server access via AWS Systems Manager (SSM) over PrivateLink
  • Stateless NACLs blocking confirmed attack flows detected in VPC Flow Logs
  • VPC endpoints for all AWS service communication — no traffic leaves your VPC for S3, KMS, Secrets Manager
  • Kubernetes: network policies deny-all by default — only explicit allow rules permitted
  • Mandatory AWS KMS CMK on all storage: S3, EBS, RDS, DynamoDB — with automatic annual rotation
  • S3 bucket policy enforcing aws:SecureTransport: true — reject all non-TLS requests
  • Kubernetes: External Secrets Operator pulling from Secrets Manager — zero secrets in YAML files
  • Enable EBS encryption by default at account level — no instance can launch with unencrypted disk
